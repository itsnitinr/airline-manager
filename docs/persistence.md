# Persistence foundation

PostgreSQL is the schema authority. Every schema change is an immutable, reviewed, forward SQL
migration in `packages/database/migrations`; production corrections use a new forward migration,
not an edited migration or an automatic rollback. The runner takes a PostgreSQL advisory lock,
checks SHA-256 checksums of applied files, applies each pending file atomically, and records it in
`schema_migrations`.

## Commands

Run these from the repository root with `MIGRATION_DATABASE_URL` or `DATABASE_URL` set:

```sh
pnpm db:migrate
pnpm db:migrate:check
pnpm db:types
pnpm db:types:check
pnpm test:database
```

Type generation introspects the applied PostgreSQL schema. The generated
`packages/database/src/generated/database.ts` file is committed; CI migrates a blank PostgreSQL
database and fails if either migrations remain pending or regeneration changes that file.

Compose runs the same migration runner as a one-shot service before API and worker startup. API,
worker, test, and migration processes use independently named, bounded pools. Pool size and timeout
environment variables contain operational settings only; connection URLs remain runtime secrets.

## Data conventions

- Persistent identifiers are PostgreSQL `uuid`, normally generated by `gen_random_uuid()`.
- Instants are `timestamptz` and database sessions are forced to UTC. `timestamp without time zone`
  is not used for instants.
- Exact money and exchange-rate values use constrained `numeric`, transported as strings; floating
  point types are not allowed for financial values. Currency-bearing domain records must include an
  explicit ISO 4217 code. Integer minor units are acceptable only when the currency scale is explicit.
- Optimistically updated rows use a positive `bigint` version. Updates compare the expected version
  and atomically increment it; a zero-row update is a concurrency conflict.
- Transactions stay short. Callers select read committed, repeatable read, or serializable isolation.
  Only PostgreSQL serialization (`40001`) and deadlock (`40P01`) failures are retried, and the whole
  callback must be safe to repeat. Constraint, timeout, connection, and application errors are never
  retried by the helper.

Domain modules own repository interfaces and domain objects. Database adapters may use generated
Kysely rows internally, but must map them at the adapter boundary; generated rows, Kysely builders,
transactions, and PostgreSQL client types are not domain API types.
